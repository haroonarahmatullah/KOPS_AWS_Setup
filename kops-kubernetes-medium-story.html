From Zero to Kubernetes: A Complete KOPS Setup Guide on AWS

Learn how to spin up a production-grade Kubernetes cluster on AWS using KOPS — from EC2 provisioning to RBAC security — with every command you actually need.

AK
Haroon · Cloud Engineer March 3, 2025 12 min read

KOPS cluster architecture: Control Plane manages state via S3 and orchestrates Worker Nodes running application Pods

Managing containers at scale is one of the most demanding challenges in modern cloud engineering. Kubernetes solves this, but getting it running on AWS from scratch? That's where most teams stumble. KOPS (Kubernetes Operations) is the tool that bridges that gap — making it straightforward to provision, manage, and evolve production Kubernetes clusters entirely from the command line.

This guide walks you through the complete journey: standing up an EC2 instance as your control machine, creating an S3-backed state store, spinning up a multi-node cluster, working with namespaces, scaling nodes, and locking things down with RBAC. Every command is real and battle-tested.

"KOPS treats your entire cluster as infrastructure-as-code — stored in S3, versioned, and recoverable at any time."

Setting Up Your Control Machine

Before touching Kubernetes, you need a bastion — a lightweight EC2 instance to run KOPS commands from. A t2.micro Amazon Linux 2 instance is all you need. Attach an IAM Role with Admin permissions to it (never hard-code credentials), and make sure the AWS CLI is available.

Installing kubectl and KOPS

Run the following on your EC2 instance to install both tools and expose them on your system path:

bash — install kubectl & kops 
# Download kubectl (latest stable)
curl -LO https://dl.k8s.io/release/$(curl -L -s \
  https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl

# Download KOPS v1.25
wget https://github.com/kubernetes/kops/releases/download/v1.25.0/kops-linux-amd64

# Make both executables and move to PATH
chmod +x kops-linux-amd64 kubectl
mv kubectl /usr/local/bin/kubectl
mv kops-linux-amd64 /usr/local/bin/kops

# Add to .bashrc (Amazon Linux only — Ubuntu skips this)
echo 'export PATH=$PATH:/usr/local/bin/' >> ~/.bashrc
source ~/.bashrc
💡

Ubuntu vs Amazon Linux: On Ubuntu, /usr/local/bin is already in $PATH by default — you can skip the .bashrc export step entirely.

Creating Your S3 State Store

S3 acts as a durable, versioned store for your entire cluster configuration

KOPS stores all cluster configuration in an S3 bucket. This is a key design decision — it means your cluster is recoverable even if nodes are destroyed, because the desired state lives outside the cluster itself. Versioning ensures you can roll back any configuration change.

bash — create & configure S3 state bucket 
# Create the bucket in ap-south-1
aws s3api create-bucket \
  --bucket haroon-kops-testbkt1234.k8s.local \
  --region ap-south-1 \
  --create-bucket-configuration LocationConstraint=ap-south-1

# Enable versioning for rollback capability
aws s3api put-bucket-versioning \
  --bucket haroon-kops-testbkt1234.k8s.local \
  --region ap-south-1 \
  --versioning-configuration Status=Enabled

# Set the env variable KOPS reads automatically
export KOPS_STATE_STORE=s3://haroon-kops-testbkt1234.k8s.local

Spinning Up Your Cluster

With the state store in place, you're ready to create the cluster. KOPS separates the declaration of a cluster from its provisioning — create cluster writes config to S3, while update cluster --yes actually provisions the AWS infrastructure.

1

Declare the cluster

Writes cluster spec to S3. Nothing is provisioned yet — this is purely configuration.

2

Provision with --yes

Creates EC2 instances, ASGs, load balancers, and networking. AWS infrastructure is live after this.

3

Validate and wait

Poll cluster health for up to 10 minutes until all nodes and components are ready.

bash — create, provision, validate 
# 1. Declare cluster (1 master c7i-flex.large, 2 workers t3.micro)
kops create cluster \
  --name haroon.k8s.local \
  --zones ap-south-1a \
  --master-count=1 --master-size c7i-flex.large \
  --node-count=2   --node-size t3.micro

# 2. Apply and grant admin kubeconfig
kops update cluster --name haroon.k8s.local --yes --admin

# 3. Wait until all nodes pass health checks
kops validate cluster --wait 10m

Once validated, check your nodes in the AWS Console — you'll see EC2 instances launched inside an Auto Scaling Group, fronted by a Classic Load Balancer (CLB). If you prefer an Application Load Balancer (ALB), edit the cluster spec and change class: Classic to class: Network.

Deploying Your First Application

With the cluster up, deploy a workload. Below is a minimal Deployment manifest running 4 replicas of a banking app image across your worker nodes. Note that the master node never schedules application pods — Kubernetes reserves it for control plane operations.

yaml — deployment.yml 
apiVersion: apps/v1
kind:       Deployment
metadata:
  name:   mb-deployment
  labels:
    app: bank
spec:
  replicas: 4
  selector:
    matchLabels:
      app: bank
  template:
    metadata:
      labels:
        app: bank
    spec:
      containers:
      - name:  cont1
        image: haroon/mb-image:latest
bash — deploy & scale 
kubectl create -f deployment.yml
kubectl get pods -o wide    # pods spread across worker nodes

# Scale out to 10 replicas
kubectl scale deploy/mb-deployment --replicas=10
# Scale back in
kubectl scale deploy/mb-deployment --replicas=4

Namespaces: Isolated Lanes for Every Team

Namespaces provide isolated environments within a single cluster — no inter-namespace visibility by default

Running a dedicated cluster for every team is expensive and operationally painful. Namespaces give you isolated, virtual clusters within a single physical cluster. Pods in the dev namespace cannot see pods in prod by default. Kubernetes ships with four built-in namespaces:

default kube-system kube-public kube-node-release

All Kubernetes control plane components — API server, scheduler, controller manager, kube-proxy — run as pods in kube-system. Application workloads go in default or custom namespaces you create.

bash — namespace workflow 
# Create and switch to dev namespace
kubectl create ns dev
kubectl config set-context --current --namespace=dev

# Launch pods in dev
kubectl run dev1 --image nginx
kubectl run dev2 --image nginx

# Create and switch to prod namespace
kubectl create ns prod
kubectl config set-context --current --namespace=prod
kubectl run prod1 --image nginx

# Cross-namespace visibility
kubectl get pods -n dev
kubectl get pods -n prod
kubectl get pods -A          # all namespaces at once

RBAC: Least-Privilege Access Control

RBAC maps Roles (permissions) to Subjects (users/service accounts) via RoleBindings

The moment your cluster is shared, you need Role-Based Access Control. RBAC answers two questions: who are you? (authentication) and what can you do? (authorization). In Kubernetes, permissions are defined in Roles, then granted via RoleBindings.

"Users are managed outside Kubernetes. Service accounts are Kubernetes-native objects — manageable with kubectl like any other resource."

Three-step RBAC setup for user "jack"

yaml — serviceaccount.yml 
apiVersion: v1
kind:       ServiceAccount
metadata:
  name:      jack
  namespace: dev
yaml — role.yml (read-only on pods) 
apiVersion: rbac.authorization.k8s.io/v1
kind:       Role
metadata:
  namespace: dev
  name:      dev-pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs:     ["get", "list", "watch"]
yaml — rolebinding.yml 
apiVersion: rbac.authorization.k8s.io/v1
kind:       RoleBinding
metadata:
  name:      read-pods
  namespace: dev
subjects:
- kind:     User
  name:     jack
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind:     Role
  name:     dev-pod-reader
  apiGroup: rbac.authorization.k8s.io

Verifying permissions with impersonation

bash — verify access 
# jack CAN list pods
kubectl auth can-i list pods --namespace=dev --as=jack
# → yes

# jack CANNOT create pods
kubectl auth can-i create pods --namespace=dev --as=jack
# → no

Cluster Admin: Scaling Nodes In and Out

KOPS uses Instance Groups (IGs) to manage node pools. Scaling out is a three-step process: edit the IG to update min/max, update the cluster to push the change, then do a rolling update.

GoalCommandNotes
List clusterskops get clusterShows all managed clusters
Edit worker IGkops edit ig --name=… nodes-ap-south-1aChange maxSize / minSize
Edit master IGkops edit ig --name=… master-ap-south-1aSame pattern
Apply changeskops update cluster --name … --yes --adminPushes config to AWS
Rolling updatekops rolling-update cluster --yesCycles nodes safely
Delete clusterkops delete cluster --name … --yesTears down all AWS resources

Troubleshooting: When Kubelet Won't Start

One of the most common issues on freshly provisioned nodes is kubelet failing to register with systemd. Here's the quick resolution path:

bash — kubelet recovery 
# Reload systemd and restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# If still missing, add a minimal service file
sudo nano /etc/systemd/system/kubelet.service

# [Service]
# ExecStart=/usr/bin/kubelet
# Restart=always
# RestartSec=10

sudo systemctl enable kubelet
sudo systemctl restart kubelet

# Tail logs to confirm
sudo journalctl -u kubelet -f

Wrapping Up

KOPS makes Kubernetes on AWS significantly less intimidating. The S3-backed state store gives you a safety net. Namespaces let you share infrastructure without chaos. RBAC closes the authorization loop. And KOPS itself handles the unsexy-but-critical work of ASG management, rolling updates, and cluster lifecycle.

The next natural steps from here are adding an Ingress controller, setting up cluster autoscaler, and integrating with AWS IAM roles for service accounts (IRSA) — but that's a story for another day.
